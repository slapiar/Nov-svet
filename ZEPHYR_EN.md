# Zephyr Project Manager as the first project-management layer of the New World system

**Document status:** working technical note  
**Working mode:** calm, careful, complete  
**Purpose:** to show how **Zephyr Project Manager** and **Zephyr Project Manager Pro** can be used as a practical first layer for testing the New World concept.

This document does not present Zephyr as the final New World system.

It presents Zephyr as the first available workshop where we can test goals, commitments, evidence, usefulness, trust, authority and sustainable cooperation without wasting energy, time, mental strength or money by building everything from scratch too early.

---

## 1. The core idea

Zephyr may be ready for a second life.

Not as another feature request. Not as a rebranding exercise. But as a new use case:

```text
project management as a practical way to test human goals, commitments, evidence, trust and sustainable cooperation
```

The New World system needs to test a very simple but deep chain:

```text
goal → promise → commitment → action → evidence → usefulness → authority → sustainable relationship
```

Zephyr Free and Zephyr Pro already contain several layers that can be mapped to this chain.

That makes Zephyr useful not only as a project manager, but as a possible testbed for a broader cooperation model.

---

## 2. The two Zephyr layers

The archive contains two WordPress plugins:

```text
zephyr-project-manager/
zephyr-project-manager-pro/
```

The free plugin provides the project-management core.

The Pro extension adds the layers that matter most for this concept:

- custom fields,
- Kanban,
- milestones,
- reports,
- frontend project access,
- REST/API possibilities,
- richer task and project workflows.

Together, these two layers are already enough to test whether the New World logic can work in practice before building a new software environment.

---

## 3. Mapping New World concepts to Zephyr

| New World concept | Zephyr layer | Meaning |
|---|---|---|
| Goal | Project | A space where an intention becomes organized |
| Promise | Task / assigned task | A concrete proposed action |
| Commitment | Task with assignee, date and status | A promise with responsibility |
| Action | Completed task / update | A visible performed step |
| Evidence | File, attachment, comment, report | Proof attached to action |
| Usefulness | Progress, result, report | The value created by the work |
| Authority | Role, owner, assignee, custom field | Permission and responsibility in context |
| Validation | Status, comment, assignment, acceptance | Confirmation, rejection or fulfilment |
| Contract | Project + tasks + files + roles + terms | A practical construction of agreement |
| Relationship | Project membership, team, discussion | The human space around the goal |

## 4. Projects as goals

In Zephyr, a project can represent a goal.

In the New World concept, a project is not just a folder of tasks. It is a living space where an idea gathers:

- people,
- roles,
- responsibilities,
- work,
- evidence,
- communication,
- progress,
- value,
- trust.

This is important because New World should not start with abstract profiles and theoretical ratings. It should start with real goals and real cooperation.

---

## 5. Tasks as promises and commitments

A Zephyr task can be used as the technical form of a promise.

If a task is assigned to a person, has a deadline, has a status and is connected to evidence, it becomes more than a task. It becomes a commitment.

```text
task without acceptance = proposal
task accepted by a member = promise
task with responsibility and time = commitment
completed task with evidence = action
validated action = stronger trust
```

This is one of the most important bridges between Zephyr and the New World system.

Authority should not grow from self-description. It should grow from accepted commitments, completed actions and visible evidence.

---

## 6. Comments as communication traces

Comments and discussions inside Zephyr can become a trace of responsibility.

A comment can be:

- an explanation,
- an acceptance,
- a refusal,
- an update,
- a warning,
- a clarification,
- a small agreement,
- a link between action and evidence.

For the New World system, communication should not float outside the goal. It should stay connected to the project, the task and the evidence.

Zephyr already supports this kind of trace.

---

## 7. Files as evidence

Files and attachments are essential.

In the New World concept, evidence can be:

- a photo,
- a document,
- a contract,
- a report,
- a delivery file,
- a screenshot,
- a field note,
- a confirmation.

This matters because trust should not grow from mood, popularity or empty claims.

Trust should grow from visible commitments and visible evidence.

---

## 8. Custom fields as the first metric layer

Zephyr Pro custom fields are especially important.

They can be used as the first simple metric layer before a dedicated New World database exists.

Possible custom fields:

- goal type,
- urgency,
- risk level,
- validation state,
- expected usefulness,
- evidence type,
- responsibility level,
- authority context,
- acceptance state,
- field action required.

This is not the full New World metric system yet.

But it allows us to test early questions without building a full custom platform too soon.

---

## 9. Milestones as development gates

Milestones can represent the development stages of a goal.

Example:

```text
idea → open goal → pilot → first evidence → first usefulness → stable process
```

A milestone is not just a deadline.

In this model, it can become a gate: the system can ask whether the goal has earned the right to move into the next stage.

---

## 10. Kanban as visible movement of commitments

Kanban can show the movement of commitments through practical states:

```text
proposed → accepted → in progress → waiting for evidence → completed → validated
```

This visual layer is useful because people do not need to read a database to understand what is happening.

They can see where a commitment stands.

---

## 11. Reports as the first picture of usefulness

Reports can show:

- what was promised,
- what was completed,
- what was delayed,
- what was supported by evidence,
- where cooperation is working,
- where cooperation is failing,
- where the process needs correction.

In the New World concept, a report should not be a tool of control over people.

It should be a tool for seeing the truth of a goal.

---

## 12. Frontend access for real members

The frontend layer matters because the New World system cannot live only inside `wp-admin`.

A normal member should be able to:

- see the goal,
- see their task,
- accept or reject responsibility,
- add a comment,
- upload evidence,
- update progress,
- follow the result.

Zephyr Pro already gives a path toward that kind of access.

---

## 13. Why the mobile app matters

The mobile app matters a lot here.

It means Zephyr can follow real people in real situations, not just project managers sitting inside `wp-admin`.

With the mobile app, Zephyr already has something many abstract cooperation systems never reach:

```text
a bridge between planning and real-world action
```

For the New World concept, this is essential.

Commitments, updates, evidence, photos, comments and progress can be captured close to the real action.

That turns Zephyr from a planning tool into a field tool.

And a field tool is exactly what is needed if trust and authority should grow from real commitments, real actions and real evidence.

---

## 14. REST/API as an integration bridge

The REST/API layer matters because Zephyr does not have to become the final architecture.

It can become the first project-management layer connected later to:

- member profiles,
- metric databases,
- validation workflows,
- authority calculations,
- evidence archives,
- notifications,
- AI guidance,
- external tools.

This keeps the architecture open.

Zephyr can be used now, while the deeper system is still being discovered.

---

## 15. What Zephyr is not yet

Zephyr is not yet:

- a full authority metric,
- a relationship operating system,
- a dynamic member profile,
- a moral-matrix database,
- a certification system,
- a complete New World platform.

That is not a weakness.

It is exactly why Zephyr is useful as a first workshop.

We can test what already works before inventing what does not exist yet.

---

## 16. What Zephyr can do immediately

Zephyr can already help us:

- create project spaces for goals,
- assign commitments to people,
- track responsibility,
- attach evidence,
- keep communication near the work,
- use custom fields,
- define milestones,
- visualize workflow with Kanban,
- produce reports,
- test frontend access,
- connect through API later,
- bring mobile field updates into the process.

That is enough for the first serious pilot.

---

## 17. Possible first pilot

A first pilot can be very small.

Example:

```text
Pilot: testing goal, commitment, evidence and usefulness in Zephyr
```

Tasks:

1. define one clear goal,
2. invite a small number of participants,
3. assign first commitments,
4. attach evidence to completed actions,
5. review usefulness,
6. write down where Zephyr works,
7. write down where Zephyr reaches its limits.

The goal of the pilot is not to prove that Zephyr is the final solution.

The goal is to discover how much of the New World logic Zephyr can already carry.
